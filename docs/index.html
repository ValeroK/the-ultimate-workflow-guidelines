The Ultimate Workflow Guidelines

A two-skill plugin for Claude Code, Cursor, and Claude Desktop that makes LLM coding assistants plan first, test first, respect existing design, and accumulate per-repo knowledge across sessions.

Install for Claude Code Install for Cursor Install for Claude Desktop GitHub repo →

The problem

LLM coding assistants are fast but forgetful. They assume when they should ask, write 200 lines when 50 would do, touch unrelated code "while they're in there," and re-derive the same project quirks every session.

"The models make wrong assumptions on your behalf and just run along with them without checking. They don't manage their confusion, don't seek clarifications, don't surface inconsistencies, don't present tradeoffs, don't push back when they should."
— Andrej Karpathy

The behavioral principles in this plugin are derived from those observations. The plan-first workflow, greenfield bootstrap, and per-repo memory pattern are layered on top — operationalizing the principles so the model actually applies them, every turn.

The two workflows

the-ultimate-workflow-guidelines

Daily feature work in an existing codebase. Any non-trivial change — new feature, bug fix, refactor.

Produces:

  • A PLAN-<feature>.md on disk with goal, files explored, existing-design review, deviation justification.
  • A test plan you confirm before code is written.
  • A minimal implementation that respects existing patterns.
  • Updates to progress.md, ## Gotchas, and memory/ after landing.

project-bootstrap-guidelines

Greenfield only. From a blank slate to a project that's ready for feature work.

Produces:

  • PRD.md (with Mermaid flow diagram + framework choices and alternatives documented).
  • CLAUDE.md, ROADMAP.md, progress.md, an empty memory.md.
  • A clean hand-off to the workflow skill once the first feature is scoped.

The behavioral layer

Both skills apply four principles before any procedural step:

🧠
Think Before Coding — surface assumptions; ask when unclear; name tradeoffs. Don't pick silently when ambiguity exists.
✂️
Simplicity First — minimum code that solves the problem. No features beyond what was asked, no abstractions for single-use code.
🔬
Surgical Changes — touch only what the request requires. No "while I was there" refactors, no formatting drift.
🎯
Goal-Driven Execution — define verifiable success criteria; loop until they pass. Tests-first turns "fix the bug" into "write the test that reproduces it, then make it pass."

The workflow is how to work; the principles are how to think.

Key files and why we need them

Each file in the repo plays a specific role. The two layers — always-loaded vs. lazy-loaded — keep the prompt slim while letting knowledge compound across sessions.

CLAUDE.md

Always-on behavioral guidance. Auto-loaded by Claude Code on every turn. Mirrors skill 1's body so the workflow + principles + memory protocol reach Claude even outside the plugin context. Why we need it: keeps the model aligned without per-session reminders.

memory.md + memory/<topic>.md

Slim index + lazy-loaded topical knowledge. memory.md is a list of one-line pointers, each with a "Read when…" cue (concrete keywords + file globs). The actual content lives in memory/<topic>.md and is only pulled into context when the current task touches the topic. Why we need it: compounds explanatory knowledge across sessions ("here's how auth works", "here's why we picked Postgres") without paying the always-loaded tax.

## Gotchas (in CLAUDE.md)

Defensive warnings, always loaded. Short rules ("Beware:…", "Don't…") that protect future sessions from non-obvious footguns. Why split from memory: different purpose. Gotchas alert; memory explains.

PRD.md (greenfield)

What this project does and why. Source of truth for product questions: users, goals, non-goals, success metrics. Also carries every confirmed framework choice with documented alternatives. Why we need it: the workflow skill reads it before planning any later feature, so the *what* and *why* are never re-litigated.

ROADMAP.md (greenfield)

Milestones and ordering. Where the project is heading; what task closes which milestone. Why we need it: persistent project-level scope, distinct from per-feature plans.

progress.md (greenfield)

Time-ordered log. What changed when, what was decided, what bug surfaced. Why we need it: session-to-session continuity for the working state of the project.

PLAN-<feature>.md (transient, per feature)

The plan for one specific change. Created at workflow step 2; survives context compaction; retired after the feature merges. Why we need it: a written artifact survives a compacted chat; chat threads don't.

rules/*.mdc

Cursor mirror of the skills. Same body content, Cursor-shaped frontmatter (alwaysApply: true for skill 1; false for skill 2 since it's greenfield-only). Why we need it: Cursor doesn't auto-load CLAUDE.md; the rule files are the per-turn entry point in that ecosystem.

skills/*/references/*.md

Templates so the model never has to invent shapes. Plan template, PRD template, ROADMAP template, progress template, project CLAUDE.md template, memory template. Why we need it: consistent output across sessions, and the workflow skill points at them by reference instead of inlining the shape every time.

Gotchas vs Memory — the real distinction

Length is a consequence, not the criterion. The two stores serve different purposes and have different shapes:

## Gotchas (CLAUDE.md)memory/<topic>.md
PurposeDefensive warningExplanatory knowledge
Voice"Beware…" / "Don't…""Here's how…" / "Here's why…"
PostureReactive — alerts to a footgunProactive — builds understanding
Always loaded?Yes — short, in CLAUDE.mdNo — Read on demand

Decision test:

  • Phrasable as "Beware:" / "Don't…"? → Gotcha (CLAUDE.md ## Gotchas).
  • Phrasable as "Here's how X works" / "Here's why we picked Y"? → Memory (memory/<topic>.md).

Worked examples:

LessonGoes where
"Tests need a live Postgres — run docker compose up -d db first."Gotcha (defensive, one-line)
"Imports use @/ aliases, not relative paths."Gotcha (defensive footgun)

"Schema: UUID v7 for PKs because…; migrations forward-only because…"memory/db.md (explanatory, multi-decision)

Install

Claude Code (plugin)

/plugin marketplace add ValeroK/the-ultimate-workflow-guidelines
/plugin install the-ultimate-workflow-guidelines

Cursor

/add-plugin ValeroK/the-ultimate-workflow-guidelines

Or browse Cursor Marketplace once the listing is live. For an offline install, download the-ultimate-workflow-guidelines-plugin.zip from the latest release — the same ZIP serves Claude Desktop and Cursor.

Claude Desktop (claude.ai)

Requires Pro, Max, Team, or Enterprise plan with code execution enabled.

  1. Download the-ultimate-workflow-guidelines-plugin.zip from the latest release (installs both skills) — or one of the per-skill ZIPs (the-ultimate-workflow-guidelines.zip, project-bootstrap-guidelines.zip) for a one-skill install.
  2. In claude.ai, go to Settings → Features → Skills → Upload Skill and select the ZIP.

Install is per-user; each teammate uploads separately.
